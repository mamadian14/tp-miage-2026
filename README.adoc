### Exercice 2 - Refactoring en architecture hexagonale
_Temps estimé : 1 h 20_

* Partir du projet `todolist-debut-ex2`

NOTE: Le projet a été réusiné suivant les principes de l'architecture hexagonale : 

image::images/archi_hexagonale.png[]
Source : http://leanpub.com/get-your-hands-dirty-on-clean-architecture[Tom Hombergs]

* Nous avons découpé le coeur en deux couches : 
  - la couche `application` qui contient tous les contrats : ports (interfaces) et les implémentations des ports d'entrée (ou "use case") et qui servent à orchestrer les entités.
  - la couche `domain` qui contient les entités (au sens DDD, pas au sens JPA). En général des classes complexes (méthodes riches, relations entre les entités)

*Rappeler en quelques lignes les grands principes de l'architecture hexagonale.*
L’architecture hexagonale (aussi appelée Ports and Adapters)
vise à séparer le cœur métier des technologies externes. Elle organise l’application autour d’un cœur métier indépendant et de ports qui
définissent les interactions avec l’extérieur.
Les principaux principes sont : Isolation du domaine métier : la logique métier ne dépend pas des frameworks, de la base de données ou de
l’interface utilisateur. Ports (interfaces) : ils définissent les opérations que l’application expose ou utilise. Adapters : ils implémentent ces ports
pour connecter l’application aux technologies externes (REST, base de données, etc.). Inversion de dépendance : les dépendances pointent ver
le domaine et non vers l’infrastructure 

Compléter ce code avec une fonctionnalité de création de `TodoItem`  persisté en base et appelé depuis un endpoint REST `POST /todos` qui :
fait 
La création d’un TodoItem est définie par le port d’entrée AddTodoItem. Ce port représente le cas d’usage permettant d’ajouter un élément dans la liste.
Le service AddTodoItemService implémente ce port et orchestre l’opération. Il utilise le port de sortie UpdateTodoItem pour sauvegarder l’objet dans la base de données.
EXEMPLE
## @Component
public class AddTodoItemService implements AddTodoItem {

    private UpdateTodoItem updateTodoItem;

    @Inject
    public AddTodoItemService(UpdateTodoItem updateTodoItem) {
        this.updateTodoItem = updateTodoItem;
    }

    @Override
    public void addTodoItem(TodoItem item) {
        updateTodoItem.save(item);
    }
}

* prend un `TodoItem` au format JSON dans le body (voir exemple de contenu plus haut);
* renvoie un code `201` en cas de succès. 
EXEMPLE 
## @PostMapping("/todos")
@ResponseStatus(HttpStatus.CREATED)
public void createTodo(@RequestBody TodoItem item) {
    addTodoItem.addTodoItem(item);
}
La fonctionnalité à implémenter est contractualisée par le port d'entrée `AddTodoItem`.

*Quels types de tests devra-t-on écrire pour les adaptateurs ?* 
Les adaptateurs nécessitent des tests d’intégration, car ils dépendent de composants externes comme le framework web ou la base de données.
Les adaptateurs REST sont testés avec des tests d’intégration web (ex : @WebMvcTest)
Les adaptateurs de persistance sont testés avec des tests d’intégration avec la base de données
Ces tests permettent de vérifier le bon fonctionnement des interactions entre l’application et les systèmes externes.

*S'il vous reste du temps, écrire quelques-uns de ces types de test.*ON RENCONTRE beaucoup d'erreur en realisant ces tests
